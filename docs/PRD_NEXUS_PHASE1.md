## 第一章：产品概述与愿景

### 1.1 产品背景

Nexus 是一款基于图谱思维的知识管理系统，当前已具备完整的 Web 端能力，包括节点编辑、超级标签、多视图展示（看板/表格/列表）、AI 副驾驶等核心功能。

随着用户场景的深化，我们观察到两个核心痛点：

| 痛点 | 现象 | 根因 |
|------|------|------|
| **碎片化信息难以捕获** | 用户在移动场景下无法快速记录闪念、语音备忘 | 缺乏移动端入口 |
| **非结构化输入难以检索** | 大量意识流文本堆积，形成「数字垃圾」 | 缺乏智能结构化能力 |

### 1.2 产品愿景

**「让正确的设备做正确的事，让输入端极致自由、展现端极致结构」**

我们将围绕两大支柱构建下一代 Nexus：

```
┌─────────────────────────────────────────────────────────────┐
│                     Nexus Phase 1 愿景                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────┐           ┌─────────────────┐        │
│   │   多端协同       │           │   AI 智能结构化  │        │
│   │   (设备协作)     │    ×      │   (内容理解)     │        │
│   └─────────────────┘           └─────────────────┘        │
│           │                             │                   │
│           ▼                             ▼                   │
│   移动端：捕获+查阅+搜索        双轨模式：主动 + 无感        │
│   桌面端：知识中枢脑            AI 指令节点：伴随式 AI       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 核心目标

#### 1.3.1 业务目标

| 目标 | 关键结果 (KR) | 衡量方式 |
|------|--------------|----------|
| **提升记录效率** | 移动端闪记到云端 < 3 秒 | 端到端延迟监控 |
| **提升信息密度** | 长文本结构化后节点数量 ≥ 3 | AI 解析结果统计 |
| **提升检索效率** | 结构化内容检索命中率提升 50% | A/B 测试对比 |

#### 1.3.2 用户体验目标

| 场景 | 体验目标 | 设计原则 |
|------|----------|----------|
| 移动端捕获 | 点击捕获 → 选类型 → 输入 → 完成，4 步内完成 | 零摩擦 |
| 移动端笔记 | 只读浏览，与 Web 端展示一致 | 信息对齐 |
| 移动端搜索 | 自然语言查询，快速定位已有内容 | 高效检索 |
| AI 重塑 | 无感完成，结果可预期、可回退 | 用户掌控感 |
| 双面硬币 | 一键切换，原文与结构化并存 | 信息不丢失 |

### 1.4 第一阶段范围边界

#### ✅ 必做 (Must Have)

| 能力 | 说明 | 端 |
|------|------|-----|
| **移动端捕获功能** | 文字、文件、拍照、语音多模态快速输入 | Mobile |
| **移动端笔记查阅** | 每日笔记和笔记本只读浏览 | Mobile |
| **移动端自然语言搜索** | 语义搜索已有节点，不产生新节点 | Mobile |
| **桌面端打包** | 将 Web 端打包为 PC/MAC 应用 | Desktop |
| **主动内联重塑** | 用户主动触发，幽灵预览 + 一键固化 | Desktop/Web |
| **后台无感重塑** | 闲置超时自动触发，平滑形变 | All |
| **双面硬币视图** | 节点级 [结构化 \| 原文] 切换 | All |
| **AI 指令节点** | 创建期/运行期/迭代期完整交互 | Desktop/Web |

#### ⏳ 二期规划 (Phase 2)

| 能力 | 说明 |
|------|------|
| **移动端轻量看板** | #todo 等核心标签的状态流转视图 |
| **移动端状态操作** | 滑动切换任务状态等交互操作 |
| **移动端笔记编辑** | 支持在移动端编辑笔记内容 |
| **移动端 AI 指令节点** | 移动端配置和管理 AI 指令节点 |

#### ❌ 不做 (Out of Scope)

| 能力 | 原因 |
|------|------|
| 高阶视图增强（看板/表格） | 优先级低，后续迭代 |
| 超级标签配置中心 | 优先级低，后续迭代 |
| 移动端完整大纲操作 | 违背「移动端轻量化」原则 |
| 多人协作编辑 | 复杂度高，需独立规划 |

### 1.5 成功指标

| 指标类型 | 指标名称 | 目标值 | 统计周期 |
|----------|----------|--------|----------|
| **活跃度** | 移动端 DAU | 上线 30 天后 ≥ 500 | 日 |
| **功能使用** | AI 重塑触发次数 | 人均 ≥ 3 次/周 | 周 |
| **满意度** | 双面硬币功能 NPS | ≥ 40 | 季度 |
| **技术** | 重塑成功率 | ≥ 95% | 实时 |

### 3.3 模式二：后台无感重塑

| **触发时机** | 光标离开节点 **或** 闲置超时 |
| **闲置超时** | 系统根据用户习惯动态计算（初始值 5 秒） |
| **内容阈值** | 节点内容 ≥ 50 字 |

#### 3.3.2 闲置超时算法

```
闲置超时 = 基准值 × 用户系数

基准值 = 5 秒（可通过后台配置调整）
用户系数 = 基于历史行为计算：
  - 如果用户经常在 3 秒内继续输入 → 系数 1.5
  - 如果用户通常间隔 10 秒以上 → 系数 0.8
```

> **注意**：闲置超时不对用户开放配置，由系统自动优化。

## 第四章：双面硬币视图交互规范

### 4.1 设计理念

**「信息不丢失，结构可回退」**

双面硬币视图解决的核心问题是：AI 结构化后，用户如何保留对原始内容的访问权？通过节点级别的 [结构化 | 原文] 切换，用户可以在任意时刻查看原始输入，甚至基于原文进行二次编辑。

### 4.2 视图状态定义

#### 4.2.1 节点视图状态

每个支持双面硬币的节点都维护以下状态：

```typescript

  displayMode: 'structured' | 'original';

  isDualViewEnabled: boolean;
  
  /** 原文快照 */
  originalSnapshot: {
    content: string;           // 原始文本内容
    createdAt: number;         // 输入时间戳
    mediaType: 'text' | 'voice' | 'image';  // 原始输入类型
    voiceUrl?: string;         // 语音原文件（如有）
    imageUrl?: string;         // 图片原文件（如有）

  structuredData: {
    summary: string;           // AI 生成的一句话总结

    structuredAt: number;      // 结构化时间戳
```

| 重新重塑 | 基于修改后的原文重新触发 AI | 点击「🔄 重新重塑」按钮 |
| 复制 | 复制原文到剪贴板 | 右键菜单 |

### 4.5 系统退让机制

#### 4.5.1 触发条件

当用户切换到「原文视图」并进行以下操作时，系统进入「退让」状态：

| 操作 | 系统响应 |
|------|----------|
| 修改原文内容 | 标记 `controlState = 'user_controlled'` |
| 删除原文内容 | 同上 |
| 调整原文格式 | 同上 |

#### 4.5.2 退让状态下的行为

| 行为 | 说明 |
|------|------|
| 结构化子树保留 | 不自动删除已生成的子节点 |
| 不再自动重塑 | 后台无感重塑不再触发 |
| 显示提示 | 原文视图显示「内容已修改，点击重新重塑以更新结构」 |

#### 4.5.3 恢复 AI 控制

用户点击「🔄 重新重塑」后：
1. 系统读取当前原文内容
2. 发送到后端 Agent 重新解析
3. 解析完成后，更新结构化子树
4. 恢复 `controlState = 'ai_controlled'`

### 4.6 多节点场景

#### 4.6.1 独立切换

每个被重塑的节点独立维护自己的视图状态。

```
示例：用户有两个长文本节点 A 和 B

节点 A: [🌟结构化] - 显示 AI 生成的子树
节点 B: [📝原文]   - 显示原始输入

两个节点可以同时存在不同的视图状态
```

#### 4.6.2 批量操作

| 操作 | 说明 |
|------|------|
| 全部切换到结构化 | 视图菜单 →「全部显示结构化」 |
| 全部切换到原文 | 视图菜单 →「全部显示原文」 |
| 重置视图 | 恢复所有节点到默认状态（结构化） |

### 5.5 与搜索节点的关系

AI 指令节点的「上下文来源」功能复用搜索节点的查询能力：

| 能力 | 搜索节点 | AI 指令节点 |

| 结果展示 | 列表展示 | 作为 AI 上下文 |
| 后续操作 | 手动操作 | AI 自动处理 |

**技术实现**：AI 指令节点的 `queryConfig` 直接调用搜索服务的接口，获取符合条件的节点列表，然后将节点内容组装为 Agent 的上下文输入。

| **双面硬币 (Dual View)** | 节点同时保留结构化视图和原文视图，可切换 |
| **捕获 (Capture)** | 移动端的多模态内容录入功能 |
| **覆层菜单 (Overlay Menu)** | 点击捕获按钮后弹出的输入类型选择面板 |
| **每日笔记 (Daily Note)** | 按日期自动归档的笔记入口 |
| **自然语言搜索** | 使用自然语言描述搜索意图，系统语义理解后返回结果 |
| **AI 指令节点** | 可配置的伴随式 AI，执行用户自定义的指令 |
