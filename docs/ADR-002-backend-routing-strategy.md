# ADR-002: 后端 API 边界策略

- 状态: Accepted
- 日期: 2026-02-28
- 决策者: SOP 调度 / 架构评审

## 背景

当前项目同时存在两套 API 实现路径：

1. Next.js API Routes（已承载主要线上功能）
2. NestJS Backend（模块化结构已具备，但未完全接管）

双轨并行在短期提供灵活性，但长期会造成契约漂移、重复维护和排障复杂度提升。

## 决策

采用**分阶段边界策略**：

1. **短期（当前版本）**  
   以 Next.js API Routes 作为唯一线上写入入口，确保现有功能稳定可用。
2. **中期（迁移窗口）**  
   NestJS 作为目标统一业务服务层，逐域迁移（nodes -> supertags -> notebooks -> users）。
3. **迁移期间约束**  
   通过统一 Prisma Schema 和共享 DTO 契约保持行为一致，禁止新增双实现分叉逻辑。

## 影响

### 正向影响

- 当前功能不被中断，避免一次性切换风险。
- 明确未来演进路径，减少“长期混合态”不确定性。
- 统一数据契约后，前后端行为可验证性提升。

### 代价与约束

- 迁移窗口内需要维护最小兼容适配层。
- 新增 API 时必须先声明归属（Next.js 临时入口 / NestJS 最终入口）。

## 执行规则

1. 新增业务逻辑优先下沉到可复用服务层，不直接耦合页面层。
2. API 错误返回统一结构：`{ success, error, code, message }`。
3. 新增核心端点需附带对应测试（单测 + 关键路径集成测试）。
